# Mosquitto Chart Design

## Scope

This chart deploys Eclipse Mosquitto as a Kubernetes-native MQTT broker using the
official `docker.io/library/eclipse-mosquitto` image. It targets lightweight IoT,
edge, test, and internal messaging workloads that need explicit listener,
authentication, ACL, TLS, and WebSocket contracts without hiding Mosquitto's
state model.

## Workload Model

- The broker runs as a StatefulSet so each pod keeps a stable DNS name and, when
  persistence is enabled, its own PVC under `/mosquitto/data`.
- `architecture.mode=standalone` is the default and requires one broker replica.
- `architecture.mode=federated` uses Mosquitto bridge configuration generated by
  the init container. Each broker connects to its peers through StatefulSet
  headless DNS and exchanges topics according to `broker.federation.*`.
- Federated mode is not presented as a native shared-state MQTT cluster. Sessions,
  retained messages, and local persistence remain broker-local unless Mosquitto
  bridge behavior propagates the relevant topic traffic.

## Networking

- The primary Service exposes MQTT, WebSocket, or MQTTS ports depending on TLS
  settings and listener toggles.
- The headless Service supports stable peer discovery for bridge generation and
  direct broker addressing.
- WebSocket Ingress is intentionally separate from MQTT TCP exposure because
  Kubernetes Ingress normally routes HTTP/WebSocket traffic, not raw MQTT TCP.
- MQTTX Web is an optional companion Deployment and Service for browser-based
  testing. It is not required for production broker traffic.

## Security Model

- Authentication is opt-in. When enabled, a password file is generated at runtime
  from either chart-managed credentials or `auth.existingSecret`.
- ACL enforcement is opt-in and can use chart-rendered ACL rules or an existing
  ConfigMap.
- TLS and mTLS use an existing Kubernetes Secret mounted into the broker pod.
  The chart does not mint certificates.
- The default anonymous broker is suitable only for trusted internal networks or
  quick evaluation. Public exposure should enable authentication, ACLs, and TLS
  or mTLS.

## Operational Contracts

- `broker.tls.enabled=true` disables the plain MQTT listener and exposes MQTTS.
- TLS mode is blocked with federated mode because the bridge generator currently
  emits plain peer connection settings.
- Multi-replica federated installs can apply default anti-affinity and topology
  spread constraints unless the operator supplies explicit placement rules.
- NodePort values are omitted when set to `0`, allowing Kubernetes to assign
  ports. Non-zero values pin listener-specific NodePorts.
- `mqttxWeb.image.tag` defaults to empty because upstream image tagging is not
  aligned enough to pin a strict tag in the chart default.

## Non-Goals

- No native MQTT cluster abstraction is implied.
- No certificate management is bundled.
- No External Secrets integration is rendered by this chart; use
  `auth.existingSecret` with an externally synchronized Secret when needed.
- No backup CronJob is provided because Mosquitto persistence is broker-local
  data and should be backed up through the platform storage layer.
